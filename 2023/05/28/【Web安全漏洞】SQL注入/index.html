产生原因

SQL注入是服务器在处理SQL语句的时候错误的拼接了用户提交的参数，打破了原有的SQL语句逻辑，导致攻击者可以部分或完全的掌控SQL语句执行效果的一类漏洞。也可以说是WEB应用程序对用户输入的数据过于信任，没有过滤或者过滤的不严谨，并且把用户输入的数据当作SQL语句带入到数据中去执行。

SQL注入漏洞诞生于静态网页转向动态网站的年代，早期Web安全的前辈无一例外都与SQL注入结下了不解之缘，也是入门Web安全的第一个门槛。

以一条SQL语句为例，目的是查询id=1的新闻页面，如果不存在就会产生异常。

但是如果做一些改动，加入and 1 = 1后，这条语句就永远为True，失去了判断的作用。

漏洞原理

SQL注入的分类

从反馈类型：有回显；无回显

从注入手法：联合注入；堆叠注入；报错注入；盲注

闭合与注释

闭合符号

以PHP的查询为例，id=1的字符1就是传入的一个变量。

如果想进行注入，首先就是去破坏原有的SQL语句结构，常见的一个检测是否有漏洞的payload就是id=1'。

输入的单引号就会和原始语句的单引号形成闭合，这样就多了一个单引号，这时候服务器就会返回一个报错，有这个报错就证明存在SQL注入。

闭合的符号并不固定，这要根据后端的拼接符号进行选择。

通常有以下这些

注释符

如果想达到目的，就必须使用注释符把这个多余的单引号处理掉。

GET和POST的注释符号是不同的，因为使用GET，数据会被进行URL编码，所以GET的注释符是--+，在被编码后会变成--空格，POST不会进行编码，使用#作为注释符。

万能密码

早期由于很多开发者对于系统的设计不够谨慎，导致出现了通过构造特殊SQL语句形成万能密码的情况。

例如一个语句通过输入的username和passwd字符验证登录信息。

假如输入的username为admin，输入的passwd为1' or '1' = '1，这样密码的条件就永远为TRUE，就可以登录任意用户名的账号，形成万能密码。

万能密码还有很多的变种

序号 万能密码
1 a’ or ‘a’ = ‘a
2 a’ or 1#
3 ‘or 1 = 1 or’
4 a” or “a” = “a

注入流程

数据库系统表

如果想进行注入，需要知道数据库的名称、表、字段等信息，找到这些信息需要依靠数据库的系统表，通过系统表获取想要的信息，不同的数据库系统表有所差距，后面所有的内容都是使用MySQL进行举例。

数据库 系统表
MySQL information_schema.tables
Oracle all_tables、user_tables
SQL Server master、sysobjects
Access 无
PostgreSQL pg_database、pg_tables
DB2 sysibm

有回显型注入

Union注入

找到注入点

通过报错信息判断后端使用的是',",)'…的闭合方式，选取合适的进行测试，如果出现报错说明存在注入点。

使用union select 1,2,3...尝试，直到不出现报错则为字段数。

也可以使用order by，这个函数原本是进行排序，但是也可以测试有多少个字段数，比一个个试方便很多。

例如union select 1,2,3使用找到了字段数，并且页面中出现了2，说明第二个数据可以被替换，就可以使用union select 1,database(),3获取数据库名，有时候后端的SQL语句后面可能有其他内容导致无法生效，可以使用注释符屏蔽掉后面语句，如1' union select 1,database(),3 --+的方式。

有一种情况，如果是一个新闻页面，使用http://exam.com?id=1' union select 1,2,3 --+的方式不会有任何数据回显，是因为该页面返回了一开始id=1的页面，这时候可以设置一个不存在的id，http://exam.com?id=-1' union select 1,2,3 --+就可以看到回显。

Dnslog带外注入

为什么使用dns盲注？

不论是bool型盲注还是时间型盲注，都需要频繁的跑请求才能够获取数据库中的值，在现代WAF的防护下，很可能导致IP被ban。我们可以结合DNSLOG快速的将数据取出。

如遇到MySql的盲注时，可以利用内置函数load_file()来完成DNSLOG。load_file()不仅能够加载本地文件，同时也能对诸如\\\\www.test.com这样的URL发起请求。

通过拼接的方式，例如database().git12.dnslog.cn 的拼接方式，在dnslog的记录中会查看到请求security.git12.dnslog.cn的记录；当然这里的security只是该数据库名称，并不是固定的。

在域名的后面，我们需要拼接一个文件名，这是因为load_file函数只能请求文件，如果不加后面的文件名，同样无法得到显示。

但是DNS带外注入只能由于WINDOWS系统，因为load_file这个函数的主要目的还是读取本地的文件，所以我们在拼接的时候需要在前面加上两个斜杠，这两个斜杠的目的是为了使用load_file可以查询的unc路径。但是Linux服务器没有unc路径，也就无法使用dnslog注入。

Payload

原理是load_file通过请求security.gitw7c.dnslog.cn下的/abc文件时留下dns解析记录来获取信息的。

解释：security.gitw7c.dnslog.cn去gitw7c.dnslog.cn中去解析的时候，就会留下解析记录，通过获取前来解析的低级域名前缀来获取我们需要的信息

SMB带外注入

与DNS带外的区别是装载数据的区域不一定要使用前缀，也可以使用后面的路径，可以在xxx.com和abc中间，这样的话就不依赖于DNS解析。Linux下可以使用Samba，在开启访问日志后，在日志中寻找带外的数据。

对抗转义的攻击方法

宽字节注入

原理是通过提交宽字节（如GBK）的半个字符，利用这半个字符和转义后的转义符’\‘进行结合，让转义符失效留下单独的引号。

注入条件：服务器脚本连接数据库时使用的是宽字节。

转义原理，使用转义符’\‘将单引号变成了一个字符串，使其SQL语句无法识别。

提交1%df'的时候语句就变成，\URL编码后为%5C，'编码后为%27

%df%5C则会组成GBK中的連，语句就变成了下面语句，导致转义失效。

二次注入

常见于白盒代码审计，黑盒很难发现。

大多数数据库会对取出的数据进行反转义变成初始的数据，漏洞来源于对自己数据库的数据信任，将数据取出反转义后导致漏洞的产生。
